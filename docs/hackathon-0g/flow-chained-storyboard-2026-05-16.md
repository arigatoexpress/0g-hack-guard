# 0guard Flow Chained Storyboard

Updated: May 16, 2026.

Purpose: build a professional Flow/Veo scene chain that explains 0guard instead
of only looking cool. Generated clips must stay textless. Real product screen
capture and narration carry exact labels, routes, addresses, hashes, and proof
claims.

## Credit Discipline

- Generate one output per prompt while iterating.
- Use the fastest/lowest-cost Veo 3.1 option for drafts when available.
- Download 720p original for review; upscale only after a scene survives QA.
- Use Extend only when the previous clip already explains the right idea.
- Reject quickly if a clip has generated text, fake UI, fake wallet approvals,
  token charts, logos, QR codes, or generic crypto hype.

## What Failed In The First Clean Clip

`0guard-flow-intent-checkpoint-clean.mp4` is visually usable as an abstract
transition, but it does not explain the product by itself. It should not be the
main explanatory scene. It can appear for one to two seconds only if the final
edit needs an atmospheric checkpoint transition.

## Story Rule

Each generated scene must answer one viewer question:

| Scene | Viewer Question | Visual Answer |
| --- | --- | --- |
| 1 | What is being protected? | A wallet vault stays closed while an agent request approaches. |
| 2 | Where does 0guard sit? | The request stops at a checkpoint before the wallet. |
| 3 | What does it inspect? | Prompt context, calldata shape, policy, and incident signals become separate scan layers. |
| 4 | What does deny mean? | Red risky flow is diverted before it can touch the wallet. |
| 5 | What does allow mean? | A safe read-only simulation passes into a receipt lane without opening the wallet. |
| 6 | Why is this real proof? | The decision becomes a tamper-evident receipt object and anchors to a 0G proof grid. |
| 7 | How does Telegram fit? | A phone receives one quality-gated alert while spam/noise dissolves before reaching it. |
| 8 | How does cross-chain fit? | Multiple native lanes converge on the checkpoint without any bridge or transfer executing. |

## Global Prompt Block

Paste this into every generated scene:

```text
No generated text anywhere. No captions, subtitles, lower thirds, title cards,
labels, words, letters, numbers, wallet addresses, transaction hashes, API
paths, QR codes, charts, terminal output, chat messages, fake dashboards, fake
wallet approvals, logos, brand lettering, or pseudo-readable marks. Exact
claims will appear only in narration and real product screen capture.

Style: clean professional cypherpunk security product film, matte black glass,
precise cyan and electric green proof lines, subtle red only for denied risk,
premium documentary lighting, realistic reflections, slow deliberate camera
movement, no hooded hacker imagery, no cartoon robot, no token-price visuals,
no bridge execution, no trading UI, no money movement.
```

## Chained Scene Prompts

### Scene 1 - Protected Wallet, Agent Request

```text
Create an 8-second 16:9 Flow/Veo scene. A generic AI agent is represented by a
small white light node on the left. A sealed wallet vault is represented by a
locked dark glass safe on the right. Between them is empty space, not a UI
screen. A glowing request packet moves from the agent node toward the vault but
stops before reaching it. The vault stays closed for the entire clip. Make the
visual relationship obvious: agent side, checkpoint gap, protected wallet side.
Use no text of any kind.
```

Voiceover cue: "An AI agent is about to ask a wallet for authority."

QA: approve only if the wallet/vault is readable as protected and no interface
or generated writing appears.

### Scene 2 - 0guard Checkpoint Appears

```text
Continue from the previous scene if it is strong. A transparent shield-shaped
checkpoint materializes between the agent request packet and the sealed wallet
vault. The packet pauses at the checkpoint. Thin cyan scan layers pass through
the packet: one layer for prompt context, one layer for transaction shape, one
layer for policy, and one layer for public incident intelligence. The layers
are abstract planes and particles, not labels. The wallet never opens. Use no
text of any kind.
```

Voiceover cue: "0guard sits in the moment before signing."

QA: approve only if the checkpoint clearly sits before the wallet and the scan
layers look like inspection, not random decoration.

### Scene 3 - Deny Before The Signer

```text
Continue from the checkpoint scene. Inside the paused packet, a red risky
thread separates from the safe cyan thread. The red thread tries to move toward
the wallet but is diverted down into a sealed quarantine channel before it can
touch the vault. The wallet remains closed and untouched. The safe thread waits
at the checkpoint instead of executing anything. Use no text of any kind.
```

Voiceover cue: "Risky actions are denied before the signer is asked."

QA: approve only if the blocked path is visually obvious and the clip does not
imply a signed transaction succeeded.

### Scene 4 - Read-Only Allow Path

```text
A new safe request packet arrives at the same checkpoint. The packet is cool
cyan with a small green verification pulse. Instead of opening the wallet, it
travels into a separate read-only simulation chamber and returns as a small
receipt-shaped proof object. The wallet vault remains closed in the background.
Use no text, no numbers, and no UI screen.
```

Voiceover cue: "Safe simulations can continue without moving funds."

QA: approve only if the safe path is not shown as money movement or wallet
approval.

### Scene 5 - Source Intelligence Becomes Detectors

```text
A dark analyst table contains public-source evidence represented as anonymous
paper cards and web fragments with no writing. The cards collapse into clean
source blocks, then into detector particles that flow into the 0guard
checkpoint. Red exploit shapes become structured cyan detector shapes. No real
news pages, no logos, no people, no text, no numbers, no charts.
```

Voiceover cue: "Source-linked incidents become detector coverage."

QA: approve only if evidence-to-detector transformation is readable and no
article-like text appears.

### Scene 6 - 0G Receipt Proof

```text
A decision receipt object forms from the checkpoint as a small transparent
crystal. The crystal locks into a calm decentralized proof grid made of nodes
and lines. The action is auditability only: no token transfer, no wallet
approval, no bridge, no swap. Use no text, no numbers, no hashes, no addresses,
no labels, and no logos.
```

Voiceover cue: "The verdict becomes proof people can inspect."

QA: approve only if it communicates receipt/proof, not payment or transfer.

### Scene 7 - Telegram Quality Gate

```text
A dark phone-shaped glass frame waits quietly. Many tiny noisy alert particles
approach but dissolve before reaching the phone. One high-quality alert packet
passes through a cyan quality gate and lands as a simple light card with no
writing. Do not show the Telegram app, chat usernames, message bubbles with
text, phone numbers, icons, or app logos.
```

Voiceover cue: "Telegram alerts should be opt-in and useful, not spam."

QA: approve only if the phone is generic and the no-spam quality gate is clear.

### Scene 8 - Native Cross-Chain Checkpoint

```text
Multiple abstract native network lanes approach one central 0guard checkpoint:
one proof lane, one EVM intent lane, one payment-intent lane, one order-intent
lane, one TON wallet-context lane, and one bridge-risk lane. Each lane stops at
the checkpoint for review. No lane executes a transfer, bridge, trade, or
payment. Use no logos, no chain names, no token symbols, no text, no numbers,
no charts, and no fake buttons.
```

Voiceover cue: "Cross-chain expansion stays checkpoint-first."

QA: approve only if lanes converge for review without implying asset movement.
